- New `git-pr` widget (git group): renders the pull request for HEAD's branch via the `gh` CLI. Network is opt-in and gated at two layers — the widget hides unless `options.allowNetwork: true` is set (defense in depth on the consumer end), and the actual `gh pr view --json number,url,title` invocation only fires when the snapshot loader is called with `allowPullRequest: true`. The render hot path (§1.2 N5) stays untouched: `loadGitSnapshot` is wired outside `renderFromInputs`, the `gh` lookup lives in a new `src/git/pr.ts` module clearly marked as the only file under `src/git/` that may shell out to a binary other than `git`, and any failure (`gh` missing, no PR, malformed response, network timeout — 1.5 s default) silently yields `null` so the widget hides without an error surface on the statusline. Variants in the catalogue: `number` (`#42`, default), `url`, `title`, `number-title` (`#42 feat: do the thing`). Plumbing: `src/git/pr.ts` exports `loadPullRequest` + a pure `parsePullRequestJson` helper for tests; `GitSnapshot` gains a required `pr: GitPullRequestInfo | null` field that's `null` everywhere unless explicitly populated; `LoadGitSnapshotInput` gains `allowPullRequest?: boolean` and `pullRequestTimeoutMs?: number`; `src/render/demo-fixture.ts`'s `demoGit` carries a sample PR so the editor preview can show every variant; `src/widgets/catalog.ts` adds the entry with four variants under the `git` family.
