- **Strict per-widget option validation.** `config widget add` / `replace` / `set-option` now reject an unknown option key for **every** widget (previously only variant-driven / reset-axis widgets were strict; variant-less widgets accepted arbitrary keys). The option spec (`src/widgets/families/option-spec/option-spec.ts`) now enumerates the full known option-key set for all 38 widgets — audited against every `settings.options.*` read in the widget sources — so a typo like `config widget set-option notakey foo` on a `model` widget fails with a catalogue-pointing `ConfigMutationError` and exit 2 instead of silently writing a dead option. `label` stays universal; the token accumulators' `reset` axis stays validated against `RESET_AXES`. A regression test loads every shipped template (`default` / `minimal` / `power`) through the same validator so a too-strict spec can never break a seeded config.
- **New `cost-vs-limit` widget** (tokens family, catalogue 37 → 38) — shows session spend against a user-configured budget, e.g. `$1.20/$5`, from `cost.totalUsd` + a per-widget `budget` option (USD, a positive number; no reset axis, like the other cost widgets). When spend reaches or exceeds the budget the cell signals with the theme's `danger` role (theme-driven — no hardcoded colour). Hides when `totalUsd` is absent or `budget` is missing / non-positive / not a number. Opt-in (not in any default fixture), so the render goldens are unchanged; the `widget.type` schema enum regenerates via the prebuild gen script (gate-28).
