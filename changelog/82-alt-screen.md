- The `agentline config` TUI editor now paints into the terminal's alternate-screen buffer (DEC private mode `?1049`), so opening and closing the picker / help overlay no longer leaves stacked stale copies of the editor in scrollback. Root cause: Ink's default inline-rendering mode commits the previous frame to scrollback whenever the rendered tree's measured height changes; the screenshots that motivated the editor sweep (two stacked "agentline config" headers, a residual picker that only disappeared after the next arrow key) were exactly that artefact. Fix: a new `enterAltScreen(stream = process.stdout): () => void` helper writes `\x1b[?1049h` on entry and returns a finalizer that writes `\x1b[?1049l`. `mountEditor` calls it once at startup and threads the finalizer through `inst.waitUntilExit().finally(...)` so the leave sequence fires on every exit path — save, `q`, `Esc`, exception. A `SIGINT` / `SIGTERM` hook (installed via `process.once`) runs the same finalizer before exit, so a `Ctrl-C` mid-edit doesn't leave the user's terminal stuck in the alt buffer; the listeners are removed on normal exit so we don't accumulate them across repeat invocations. The finalizer is idempotent — calling it twice doesn't double-write the leave sequence. Both halves no-op when `stream.isTTY` is `false`, so non-interactive consumers (CI runs, redirected output, vitest) are unaffected. `mountEditor`'s `render(...)` call now also passes `{ patchConsole: false, exitOnCtrlC: true }` so stray writes from third-party deps don't fight the alt-screen surface and Ink keeps owning Ctrl-C. Four new tests in `main.test.ts` cover the four contract points: TTY-off no-op, TTY-on write pair, idempotent restore, SIGINT-triggered restore. 1127 passing across 112 files; 12 gates green. The render hot path (`src/render/*`) is unchanged — `enterAltScreen` ships only from `src/tui/main.ts`, which is dynamically imported by `agentline config`, so the statusline render path picks up no new bytes.
