- fix: `reset-timer` no longer prints a confident `reset in 0m` for a rate-limit window whose `resets_at` has already elapsed. Claude Code refreshes its `rate_limits` block on API activity rather than on a timer, so around a window rollover — or during a long idle — it keeps shipping a reset instant that has passed; subtracting it from now yielded a negative remaining that the duration formatter clamped to `0m`, indefinitely. Observed on a live render as `reset in 0m · weekly 6d 23h 9m`, where the weekly window was perfectly healthy. An elapsed `resets_at` is now treated as an absent one, so the widget falls back to its local block/week derivation and shows a forward-looking countdown. Wall-clock (`at-*`) formats take the same fallback instead of rendering an instant already in the past.
